# Known Issues & Design Notes — MeshVQVAE

---

## Issue 1: Query Points Are Frozen (Never Re-Sampled)

**Severity:** Medium  
**Status:** NOT FIXED — current training run is unaffected for now  
**File:** `mesh_vqvae/src/preprocessing.py`, `mesh_vqvae/src/dataset.py`

### What is happening
`query_pts` (the 2048 3D points used by the occupancy decoder) are computed **once** during
preprocessing and saved to disk cache (`runs/cache/`). Every training step, `dataset.py`
just loads this fixed cache and returns the **exact same** query points and occupancy labels
for each mesh, every epoch forever.

```python
# dataset.py __getitem__  — query_pts come from cache, never refreshed
query_pts = data["query_pts"]   # always the same 2048 points
occupancy = data["occupancy"]   # always the same fixed labels
```

### Why this is a problem
- The model can **memorize** the fixed label pattern for each mesh rather than truly
  learning the occupancy function. This causes the `val/iou` to plateau early.
- The query points are a mix of: 50% near-surface (σ=0.05), 25% fine-surface (σ=0.01),
  25% uniform random. All of this randomness is **frozen after the first cache write**.

### Is it okay to leave unfixed for now?
**Mostly yes.** The model is still learning shape structure (IoU=0.42, cls_acc=60% at
epoch 47). The fixed query set still covers the object well enough to train. The risk
is a ~5-10% IoU ceiling lower than optimal. For a first training run this is acceptable.

## Issue 2: Query Points Not Rotation-Consistent With Surface Points

**Severity:** Medium  
**Status:** NOT FIXED  
**File:** `mesh_vqvae/src/dataset.py`, `mesh_vqvae/src/preprocessing.py`

### What is happening
During training, `augment_points` applies a random rotation + scale + jitter to the
surface points (`points`, `normals`). However, `query_pts` are pulled from the fixed
cache and are **never rotated** to match.

```python
# dataset.py __getitem__
if self.use_augmentation:
    points, normals, curvature = augment_points(points, normals, curvature)
    # ← query_pts is NOT rotated to match!

out = {
    "points":    torch.from_numpy(points),   # rotated
    "query_pts": query_pts,                  # unrotated original pose
    ...
}
```

### Why this is a problem
The encoder sees a rotated point cloud and produces rotated grid features. The decoder
is then asked to predict occupancy at query points in the **original unrotated pose**.
These are inconsistent — the encoder's 3D feature grid is rotated but the decoder query
coordinates are not.

### Is it okay to leave unfixed for now?
**Yes, with caveats.** The model appears to be learning despite this mismatch (IoU is
still improving). The decoder uses trilinear interpolation on the full 3D feature grid,
so it can partially compensate. However, this bug artificially limits reconstruction
quality and adds noise to the occupancy loss signal.

## What To Fix (When Ready)

### Fix for Issue 1 (recommended for next training run)
In `dataset.py __getitem__`, instead of loading cached `query_pts`, resample fresh
query points from the mesh on each call. This requires storing the mesh path and
calling `sample_query_points(mesh, num_query)` every step. Adds ~5-10ms I/O per sample.

### Fix for Issue 2 (minimal, no I/O cost)
In `augment_points` (`preprocessing.py`), also return the rotation matrix `R`.
In `dataset.py`, apply the same `R` to `query_pts` before returning.

```python
# Pseudocode for the fix
R, points_aug, normals_aug, curvature_aug = augment_points(...)
query_pts_aug = (query_pts.numpy() @ R.T).astype(np.float32)
out["query_pts"] = torch.from_numpy(query_pts_aug)
```
